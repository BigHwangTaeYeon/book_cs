# 프로세스와 스레드
프로세스(process) : 실행되고 있는 프로그램, CPU 스케줄링의 대상이 되는 작업(task)와 같은 의미
스레드(thread) : 프로세스의 작업 흐름

HDD/SDD에 프로그램이 올라가면 프로세스가 RAM에 올라가면서 인스턴스화가 되고,
CPU 스케줄러에 따라 CPU가 프로세스를 실행한다.

### 프로세스와 컴파일 과정
프로세스는 메모리에 프로그램이 올라가 인스턴스화가 된 것을 말한다.

소스 코드 파일 > 전처리 > 컴파일러 > 어셈블리어 > 어셈블러 > 목적코드 > 링커 > 실행 가능한 파일
                                                          라이브러리
(C언어 기준 컴파일 과정)
전처리 : 주석을 제거하고 헤더파일을 병합하여 매크로를 치환
컴파일러 : 오류처리와 코드 최적화하여 어셈블리어로 변환
어셈블리어 : 목적코드로 변환(.java > .class로 변환), BigHwang.c > BigHwang.o 파일 생성(확장자는 운영체제 마다 다름)
링커 : 라이브러리 함수와 목적 코드를 결합하여 파일 생성, .exe 또는 .out의 확장자
    - 정적 라이브러리
        빌드 시, 라이브러리가 제공하는 모든 코드를 실행 파일에 적용.
        외부 의존도가 낮은 장점과 메모리 효율성이 떨어지는 단점
    - 동적 라이브러리
        실행 시, 필요할 때만 DLL 함수 정보를 통해 라이브러리를 사용
        메모리 효율성에서 장점과 외부 의존도가 높아지는 단점

### 프로세스의 상태

생성    대기    실행    종료
흐름으로 가고 대기 상태에서 대기 중단, 중단의 상태가 더 있으며,
중단에서는 일시중단의 상태값이 있고 대기중단으로 넘어갈 수있다.

##### 생성 상태
프로세스가 생성된 상태, fork() 또는 exec() 함수로 생성되며, 이때 PCB가 할당 된다.
    - fork() : 부모 프로세스의 주소 공간을 복사하여 자식 프로세스를 생성
    - exec() : 새로운 프로세스 생성

##### 대기 상태
메모리 공간이 충분하면 할당받고, 아니면 대기하며 CPU 스케줄러로부터 CPU 소유권이 넘어오길 기다리는 상태

##### 대기 중단 상태
메모리 부족으로 일시 중단

##### 실행 상태
CPU 소유권과 메모리를 할당 받고 인스트럭션(명령, 지시)을 수행 중인 상태(CPU burst가 일어났다 라고도 표현)

##### 중단 상태
프로세스가 차단된 상태

##### 일시 중단 상태
메모리 부족으로 일시 중단

##### 종료 상태
메모리와 CPU 소유권을 모두 놓아주는 상태
부모 프로세스가 자식 프로세스를 강제시키는 비자발적 종료도 있다.
    자식 프로세스의 자원 한계치를 넘거나 부모 프로세스가 종료되었거나 사용자가 명령어로 프로세스를 종료할 때 발생

### 프로세스의 메모리 구조
스택        (동적 영역)
힙          (동적 영역)
데이터 영역 (정적 영역)
코드 영역   (정적 영역)

##### 스택과 힙
스택과 힙은 동적 할당으로 런타임 단계에서 메모리를 할당받는다.
스택 : 지역 변수, 매개변수, 실행되는 함수에 따라 할당되는 메모리 영역
    재귀 함수에서는 새로운 스택 프레임이 매번 사용되어 다른 인스턴스 변수를 방해하지 않는다.
힙 : malloc(), free() 함수를 통해 관리할 수 있으며, 동적으로 관리되는 자료 구조의 경우 힙 영역을 사용
    vector는 내부적으로 힙 영역을 사용

##### 데이터 영역과 코드 영역
정적 할당은 컴파일 단계에서 메모리를 할당한다.
데이터 영역
    BSS segment : 전역 변수 또는 static const로 선언되며 0으로 초기화 또는 초기화가 되있지 않은 변수들
    Data segment : 전역 변수 또는 tatic contst로 선언되며 0이 아닌 값으로 초기화된 변수들
코드 영역(code segment) : 프로그램 코드가 해당

### PCB(Process Control Block)
프로세스에 대한 메타데이터를 저장한 데이터, 프로세스 제어 블록이라 한다.
1) 프로그램 실행
2) 프로세스 생성
3) 스택 힙 등의 구조 기반으로 메모리 할당
4) 프로세스의 메타데이터들이 PCB에 저장되어 관리
일반 사용자가 접근 못하게 커널 스택의 가장 앞부분에서 관리
    메타데이터 : 데이터를 설명하는 작은 데이터, 대량의 정보에서 효율적으로 정보를 찾기 위해 일정한 규칙으로 부여되는 데이터 

##### PCB의 구조
- 프로세스 스케줄링 상태 : 프로세스가 CPU의 소유권을 얻은 이후 상태
- 프로세스 ID : 해당 프로세스의 자식 프로세스 ID
- 프로세스권한 : 컴퓨터 자원 또는 I/O 디바이스에 대한 권한 정보
- 프로그램 카운터 : 프로세스에서 실행할 다음 명령어의 주소에 대한 포인터
- CPU 레지스터 : 프로세스를 실행하귀 위해 저장해야하는 레지스터 정보
- CPU 스케줄링 정보 : CPU 스케줄링에 의해 중단된 정보
- 계정 정보 : 프로세스 실행에 사용된 CPU 사용량, 실행한 유저의 정보
- I/O 상태 정보 : 프로세스에 할당된 I/O 디바이스 목록 

##### 컨텍스트 스위칭(Context Switching)
PCB를 교환하는 과정, 프로세스에 할당된 시간이 끝나거나 인터럽트에 의해 발생.
1) 프로세스 A의 PCB를 저장
    ...
2) 프로세스 B의 PCB를 로드
3) 실행(인터럽트 또는 시스템콜)
4) 프로세스 B의 PCB를 저장
    ...
5) 프로세스 A의 PCB를 로드
한 프로세스의 PCB에 저장하고 다른 프로세스의 PCB를 로드하면 저장한 프로세스는 로드할 때 까지 프로세스는 유휴 시간이 발생,
이렇게 한 프로세스를 로드할 때는 다른 프로세스는 유휴 시간이 발생하며 컨텍스트 스위칭에 드는 비용이 발생한다.
또한 캐시미스로 비용이 발생하기도 한다.
    - 캐시미스 : 컨텍스트 스위칭이 일어날 때, 프로세스가 메모리 주소를 가지고 있으면 잘못된 주소 변환이 생기므로 캐시클리어 과정을 겪으며 캐시미스가 발생
    - 스레드에서의 컨텍스트 스위칭 : 스레드는 스택 영역을 제외한 모든 메모리를 공유하기에 시간과 비용이 적게 든다.

### 멀티프로세싱
두 가지 이상의 일을 수행할 수 있는 것을 지칭.
하나 이상의 일을 병렬 처리할 수 있으며, 프로세스에 문제가 발생해도 다른 프로세스로 처리할 수 있으며 신뢰성이 높은 강점을 가진다. 

##### 웹 브라우저
- 브라우저 프로세스 : 주소 표시줄, 북마크 막대, 뒤로가기 & 앞으로가기 버튼 등
    네트워크 요청이나 파일 접근 같은 권한을 담당
- 랜더러 프로세스 : 웹 사이트가 보이는 부분의 모든 것을 제어
- 플러그인 프로세스 : 웹 사이트에서 사용하는 플러그인을 제어
- GPU 프로세스 : GPU를 이용하여 화면을 그리는 부분을 제어

##### IPC(Inter Process Communication)
프로세스끼리 데이터를 주고받고, 공유 데이터를 관리하는 메커니즘
클라이언트는 데이터를 요청하고 서버는 요청에 응답하는 것도 IPC
공유 메모리, 파일, 소켓, 익명 파이프, 명명 파이프, 메시지 큐가 있다.
(메모리를 공유하는 스레드보다 속도가 떨어진다.)
- 공유메모리(shared memory) : 여러 프로세스가 동일한 메모리 블록에 접근 권한이 부여되어 프로세스끼리 서로 통신을 하는 것
    데이터 복사의 오버헤드가 발생하지 않아 가장 빠르고 동기화가 필요하다. 
- 파일 : 디스크에 저장된 데이터나 파일 서버에서 제공한 데이터를 말한다.
- 소켓 : 다른 프로세스나 네트워크 인터페이스를 통해 데이터를 전송하는 의미, TCP와 UDP가 있다.
    **TCP (전송 제어 프로토콜)**은 두 개의 호스트를 연결하고 데이터 스트림을 교환하게 해주는 중요한 네트워크 프로토콜입니다.
    **사용자 데이터그램 프로토콜(UDP)**은 통신 프로토콜로, 특히 비디오 재생 또는 DNS 조회와 같이 시간에 민감한 전송을 위해 인터넷을 통해 사용됩니다.
- 익명 파이프(unamed pipe) : 프로세스 간에 FIFO방식으로 임시 공간인 파이프를 기반으로 데이터를 주고 받으며, 단방향 읽기 전용, 쓰기 전용을 만들어 작동하는 방식
    부모 자식 프로세스 간에만 사용가능하며 다른 네트워크에서는 불가하다.
- 명명된 파이프(named pipe) : 파이프 서버와 하나 이상의 파이프 클라이언트 간의 통신을 위한 명명된 단방향 또는 양방향 파이프
    클라이언트/서버 통신을 위한 별도 파이프를 제공, 여러 파이프를 동시에 사용 가능
    프로세스끼리, 다른 네트워크 상의 통신이 가능하다.
    하나의 인스턴스를 열거나 여러개의 인스턴스를 기반으로 통신
- 메시지 큐 : 메시지를 큐 데이터 구조 형태로 관리
    공유 메모리를 통해 IPC 구현할 때, 쓰기 및 읽기 빈도가 높으면 동기화 때문에 기능을 구현하는 것이 매우 복잡해지는데, 이때 메시지 큐를 사용하기도 한다.

### 스레드와 멀티스레딩

##### 스레드
프로세스의 실행 가능한 가장 작은 단위, 한 프로세스는 여러 스레드를 가질 수 있다.
코드, 데이터, 스택, 힙 영역을 각각 생성하는 프로세스와 달리 
스레드는 코드, 데이터, 스택, 힙 영역을 스레드끼리 공유한다.
그 외의 영역은 각각 생성된다.

##### 멀티스레딩
스레드끼리 서로 자원을 고유하기에 효율성이 높다.
웹 서버의 경우 프로세스 대신 스레드를 사용하면 훨씬 적은 리소스를 소비하며 
한 스레드가 중단(blocked)되어도 다른 쓰레드가 실행(running) 상태로 중단되지 않은 빠른 처리가 가능하다.
동시성에도 큰 장점이 있다.
그러나 한 스레드에 문제가 생기면 다른 스레드에도 영향이 가서 프로세스에 영향을 줄 수도 있다.

### 공유 자원과 임계 영역

##### 공유 자원(shared resource)
시스템 안에서 각 프로세스, 스레드가 함께 접근할 수 있는 디바이스, 메모리, 데이터 등 자원이나 변수 등을 의미
두 개 이상의 프로세스가 동시에 읽거나 쓰는 상황을 경쟁 상태(race condition)이라 한다.
접근 타이밍이나 순서 등으로 결과 값에 영향을 줄 수 있는 상태.

##### 임계 영역(critical section)
둘 이상의 프로세스 스레드가 공유 자원에 접근할 때, 순서 등의 이유로 결과가 달라지는 코드 영역.
해결 방법으로 뮤텍스, 세마포어, 모니터가 있으며 상호 배제, 한정 대기, 융통성이라는 조건을 만족한다.
이 방법들의 메커니즘은 잠금(lock)이다.
    - 상호 배제 : 한 프로세스가 임계 영역으로 들어갔을 때, 다른 프로세스는 들어갈 수 없다.
    - 한정 대기 : 특정 프로세스가 영원히 임계 영역에 들어가지 못하면 안된다.
    - 융통성 : 한 프로세스가 다른 프로세스의 일을 방해해서는 안된다.
    - 뮤텍스(mutex) : 공유 자원을 lock() 설정하고 사용한 후 unlock()하는 객체, 두가지 상태만을 가진다.
    - 세마포어(semaphore) : 일반화된 뮤텍스, 간단한 정수 값과 wait(P 함수)과 signal(V 함수) 함수로 공유 자원에 대한 접근을 처리
        wait()은 자신의 차례를 기다리는 함수
        signal()은 다음 프로세스로 순서를 넘겨주는 함수
    공유 자원에 접근하면 wait() > 공유 자원을 해제하면 signal()
    - 바이너리 세마포어 : 0과 1 두가지 값만 가지는 세마포어
    - 카운팅 세마포어 : 여러 개의 값을 가지는 세마포어, 여러 자원에 대한 접근을 제어
    - 모니터 : 둘 이상의 프로세스 또는 스레드가 공유 가원에 안전하게 접근하도록 
        공유 자원을 숨기고, 접근에 대해서 인터페이스만을 제공
        세마포어보다 구현이 쉬우며, 상호배제는 자동인 반면 세마포어는 명시적으로 구현해야하는 차이점이 있다.

### 교착 상태(deadlock)
둘 이상의 프로세스들이 서로의 자원을 기다리며 중단된 상태

##### 교착 상태의 원인
- 상호 배제 : 한 프로세스가 자원을 독점하여 다른 프로세스들이 접근 불가능한 상태
- 점유 대기 : 특정 프로세스가 점유한 자원을 다른 프로세스가 요청한 상태
- 비선점 : 다른 프로세스 자원을 강제적으로 가져올 수 없음
- 환형 대기 : 한 프로세스 다른 프로세스의 자원을 요청하고 다른 프로세스는 한 프로세스의 자원을 요청한, 서로의 자원을 요청하는 상태

##### 교착 상태의 해결 방법
1. 자원 할당 시, 교착상태의 조건이 성립되지 않도록 설계
2. 교착 상태의 가능성이 없을 때만 자원 할당, 은행원 알고리즘 사용
3. 교착 상태 발생 시, 사이클이 있는지 찾아보고 관련된 프로세스를 한 개씩 삭제
4. 교착 상태가 발생하면 사용자가 작업을 종료. (현대 운영체제의 방법)
    - 은행원 알고리즘 : 총 자원의 양과 현재 할당된 자원의 양을 기준으로 안정과 불안정 상태로 나누어 
        안정 상태로 가도록 자원을 할당하는 알고리즘
